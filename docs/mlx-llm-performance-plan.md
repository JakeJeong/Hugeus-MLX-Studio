# MLX 기반 Apple Silicon 로컬 LLM 런타임 계획서

## 1. 목표

이 프로젝트의 1차 목표는 `M4 16GB Mac mini`에서 동일한 LLM 조건 기준으로 `LM Studio` 및 `Ollama`보다 더 빠르거나 더 쾌적한 로컬 추론 경험을 제공하는 것이다.

여기서 "더 빠르다"는 말은 감각적인 인상이 아니라 아래 지표로 정의한다.

- `TTFT(Time To First Token)`: 첫 토큰까지 걸리는 시간
- `Prefill throughput`: 프롬프트 입력 처리 속도
- `Decode throughput`: 토큰 생성 속도
- `Peak memory`: 최대 메모리 사용량
- `Load time`: 모델 로드 완료까지의 시간
- `Responsiveness`: 스트리밍 중 UI 및 API 응답성

## 2. 전제와 현실 체크

- `Ollama`는 실질적으로 `llama.cpp` 계열 최적화의 이점을 누린다.
- `LM Studio` 역시 범용 로컬 추론 도구로서 이미 상당한 수준의 최적화를 갖고 있다.
- 따라서 단순한 채팅 앱이나 래퍼 수준으로는 이기기 어렵다.
- 승산이 있는 영역은 `Apple Silicon 전용 최적화`, `MLX 전용 경로`, `캐시 전략`, `resident model`, `메모리 압박 제어`, `빠른 첫 응답`이다.
- 목표는 "모든 모델에서 절대 우위"가 아니라 "특정 모델군과 특정 사용 시나리오에서 명확한 우위"를 만드는 것이다.

## 3. 핵심 가설

### 가설 A. Apple Silicon 전용 최적화는 아직 공략 여지가 있다

- `MLX`는 Apple Silicon의 unified memory와 Metal 경로에 밀접하게 맞춰져 있다.
- 범용 런타임보다 `MLX 전용 설계`가 더 낮은 오버헤드를 만들 수 있다.

### 가설 B. 체감 속도는 raw tok/s 외의 요소에 크게 좌우된다

- 사용자는 토큰/초보다 `첫 응답 속도`, `모델 전환 속도`, `끊김 없는 스트리밍`, `메모리 안정성`에 민감하다.
- `TTFT`, `prefix cache`, `session KV reuse`를 잘 설계하면 체감상 더 빠른 제품이 가능하다.

### 가설 C. 동일 모델 비교에서도 승산은 있다

- 같은 모델, 같은 양자화, 같은 출력 조건에서조차 `prefill 최적화`, `KV cache 관리`, `토크나이저/템플릿 재계산 제거`, `불필요한 동기화 제거`로 일부 우위 가능성이 있다.
- 다만 이 부분은 문서 조사와 실제 벤치마크로 검증해야 한다.

### 가설 D. 범용성보다 모델 범위를 좁히는 편이 유리하다

- `M4 16GB` 환경에서는 `3B~7B` 급, 특히 `4bit` 계열에서 가장 제품성이 높다.
- 초반에는 지원 모델 수를 줄이고 해당 모델군에서 최고의 속도와 안정성을 만드는 편이 맞다.

## 4. 조사 단계에서 반드시 답해야 할 질문

### 경쟁 제품 조사

- `llama.cpp`는 Apple Silicon에서 어디서 성능을 얻고 있는가
- `Ollama`는 자체 오버헤드가 어디서 발생하는가
- `LM Studio`는 모델 로드, 캐시, UI 응답성 측면에서 어떤 장단점을 가지는가
- `mlx-lm`은 공식적으로 무엇을 이미 지원하고, 어디까지 최적화되어 있는가

### 기술 가설 검증

- MLX 경로에서 `prefill`과 `decode` 중 어디가 병목인가
- Python 오버헤드가 실제 hot path에 얼마나 영향을 주는가
- `Swift + MLX Swift` 경로가 Python 경로보다 실제로 유리한가
- `custom Metal kernel`이 필요한 병목은 어느 연산인가
- unified memory 환경에서 `resident model` 유지가 어느 정도 이득을 주는가

### 제품 전략 검증

- 우리는 `raw tok/s`를 이길 것인가, `TTFT`를 이길 것인가, 아니면 전체 UX를 이길 것인가
- 초반 타깃 모델은 무엇인가
- OpenAI 호환 API 서버를 먼저 만들 것인가, 네이티브 앱을 먼저 만들 것인가

## 5. 성공 기준

1차 성공 기준은 아래 중 하나 이상을 만족하는 것이다.

- 동일 모델, 동일 양자화, 동일 프롬프트 조건에서 `TTFT 20% 개선`
- 동일 조건에서 `decode tok/s 10% 이상 개선`
- 동일 조건에서 `peak memory 15% 이상 절감`
- 사용자가 체감할 수 있는 수준으로 모델 로드 지연과 스트리밍 끊김 감소

2차 성공 기준은 아래다.

- `OpenAI-compatible local API` 제공
- macOS 네이티브 UI 제공
- 모델 캐시, 세션 캐시, 성능 로그를 제품 수준으로 제공

## 6. 단계별 실행 계획

## Phase 0. 조사 및 기준 정의

목표:

- 현재 공개 자료를 전부 넓게 훑는 것이 아니라, 초기 벤치마크와 병목 가설을 세우는 데 필요한 정보만 수집한다.
- 비교 실험 조건을 고정한다.

산출물:

- 경쟁 제품 비교 메모
- 측정 지표 정의 문서
- 초기 타깃 모델 목록
- 벤치마크 프롬프트 세트
- baseline 측정 체크리스트

할 일:

- `MLX`와 `mlx-lm`에서 `prefill`, `decode`, `model load` 시간을 어떻게 분리 측정할지 정리
- `Ollama`와 비교 가능한 동일 모델 후보 2개 선정
- 벤치마크용 프롬프트 세트와 반복 실행 조건 정의
- Apple 공식 성능 도구 중 초기에 꼭 쓸 도구만 선정
- 비교 공정성 원칙 정의: 동일 모델, 동등 양자화, 동일 프롬프트, 동일 max tokens

완료 조건:

- `mlx-lm` 경로에서 `load`, `prefill`, `decode`, `TTFT`를 분리 측정할 수 있는 방법이 정리되어 있어야 한다.
- `Ollama`와 동일 조건으로 비교할 초기 모델 1개 이상이 확정되어 있어야 한다.
- 벤치마크 프롬프트, 출력 토큰 수, 반복 횟수가 고정되어 있어야 한다.
- "무엇과 무엇을 어떤 조건으로 비교할지"를 한 문장으로 말할 수 있어야 한다.

## Phase 1. 벤치마크용 최소 추론 엔진

목표:

- UI 없이 오직 추론 성능만 측정 가능한 최소 런타임을 만든다.

산출물:

- CLI 또는 로컬 API 서버
- 모델 로드/프롬프트 입력/토큰 스트리밍/성능 로그 기능
- 벤치마크 스크립트

핵심 기능:

- 모델 로드 시간 측정
- TTFT 측정
- prefill throughput 측정
- decode throughput 측정
- 메모리 사용량 기록
- 동일 실험 반복 실행

완료 조건:

- 같은 프롬프트 세트를 20회 이상 반복 측정할 수 있어야 한다.
- baseline 벤치마크 결과가 저장되고 재실행 시 재현 가능해야 한다.

## Phase 2. baseline 프로파일링 및 병목 확인

목표:

- 캐시나 저수준 최적화에 들어가기 전에 현재 병목을 최소 1회 정량적으로 확인한다.

산출물:

- baseline 벤치마크 결과
- `Time Profiler` 또는 동등한 CPU 프로파일 결과
- 가능하면 `Metal System Trace` 1회 결과
- 구간별 측정 포인트 정의

확인 포인트:

- `prefill`과 `decode` 중 어느 쪽이 먼저 병목인지
- Python 레벨 오버헤드가 hot path에 의미 있게 들어가는지
- 토크나이저/템플릿 처리 비용이 얼마나 되는지
- CPU-GPU 경계와 동기화 지점이 어디인지

완료 조건:

- 병목 상위 2~3개를 정량적으로 설명할 수 있어야 한다.
- 다음 최적화 단계에서 무엇을 먼저 건드릴지 우선순위가 정해져 있어야 한다.

## Phase 3. 캐시 및 런타임 최적화

목표:

- 속도에 가장 큰 영향을 주는 캐시 전략을 먼저 적용한다.

산출물:

- system prompt prefix cache
- session KV cache reuse
- tokenizer/chat template 캐시
- resident model 정책

우선순위:

- `TTFT`를 가장 먼저 줄인다.
- 그 다음 `prefill`
- 그 다음 `decode`

완료 조건:

- baseline 대비 명확한 개선 수치가 나온다.

## Phase 4. 재프로파일링 및 저수준 최적화

목표:

- 캐시 및 1차 최적화 이후 남아 있는 병목을 다시 측정하고, 저수준 최적화가 필요한 부분만 깊게 판다.

산출물:

- `Time Profiler` 결과
- `Metal System Trace` 결과
- `Memory` 분석 결과
- 구간별 `os_signpost` instrumentation

확인 포인트:

- 불필요한 `eval()` 및 동기화 지점
- CPU-GPU 경계에서의 병목
- sampling 단계의 직렬 처리 비용
- KV cache 접근 패턴
- 작은 커널 다발 실행으로 인한 오버헤드

완료 조건:

- 병목 상위 3개를 정량적으로 설명할 수 있어야 한다.

## Phase 5. Swift 네이티브 제품화

목표:

- 빠른 엔진을 macOS 제품 경험으로 연결한다.

산출물:

- `SwiftUI` 기반 데스크탑 앱
- 모델 관리 UI
- 스트리밍 채팅 UI
- 성능 로그 뷰
- 로컬 OpenAI 호환 API 옵션

완료 조건:

- 개발자용 벤치마크 도구와 사용자용 앱이 분리되어 운영 가능해야 한다.
- Python 경로 대비 Swift 전환 이유가 성능 또는 제품 통합 측면에서 문서화되어 있어야 한다.

## 7. 권장 기술 스택

### 초기 연구/실험

- `MLX`
- `mlx-lm`
- `Python`
- `FastAPI` 또는 CLI

이유:

- 가장 빠르게 가설 검증 가능
- 공식 예제와 문서 활용이 쉬움
- 벤치마크 자동화가 쉬움

### 제품화 단계

- `Swift`
- `SwiftUI`
- `mlx-swift` 또는 `mlx-swift-lm`
- 필요 시 `Metal` custom kernel

이유:

- macOS 통합성
- 메모리/성능 추적 용이성
- 제품 반응성 향상 가능

### Python에서 Swift로 전환하는 기준

- Python 경로에서 1차 성능 목표를 달성했고, 이후 제품 통합성과 배포 경험 개선이 더 중요해진 경우
- 또는 baseline/reprofile 결과에서 Python 오버헤드가 실제 hot path 병목으로 확인된 경우
- 또는 macOS 네이티브 메모리 관리, Instruments 연동, 앱 반응성 제어가 핵심 요구사항이 된 경우
- 반대로 위 조건이 충족되지 않으면 Python 경로를 유지하고 최적화를 계속한다

## 8. 초기 비교 대상

초기 비교 대상은 아래처럼 단순화한다.

- 비교 엔진 1: `mlx-lm` 기반 자체 런타임
- 비교 엔진 2: `Ollama`
- 비교 엔진 3: 가능하면 `LM Studio`

초기 모델 후보:

- `3B`급 instruct 모델 1개
- `7B`급 instruct 모델 1개

양자화 원칙:

- 되도록 `4bit` 중심
- 비교 시 가능한 한 같은 포맷/동등 수준 양자화 사용

## 9. 주요 리스크

- `llama.cpp` Metal 경로가 이미 매우 빨라 raw decode에서 우위를 만들기 어려울 수 있다.
- MLX Python 경로는 구현이 빠르지만 hot path 오버헤드가 있을 수 있다.
- `LM Studio`는 내부 구현 비교가 어려워 정밀한 원인 분석이 제한될 수 있다.
- 16GB 메모리 환경에서는 모델 크기와 컨텍스트 길이의 타협이 필수다.
- "모든 모델 지원" 욕심이 들어오면 최적화 목표가 흐려질 수 있다.

## 10. 의사결정 원칙

- 범용성보다 속도 우선
- 모델 수보다 완성도 우선
- 앱 UI보다 벤치마크 엔진 우선
- 감보다 프로파일링 우선
- 대형 모델보다 `3B~7B` 실사용 모델 우선

## 13. 최종 한 줄 전략

`M4 16GB`에서 모든 모델을 가장 잘 돌리는 범용 런처를 만드는 것이 아니라, `소수의 실사용 모델을 Apple Silicon에 맞게 가장 빠르고 매끄럽게 돌리는 MLX 기반 런타임`을 만드는 것이 이 프로젝트의 핵심 전략이다.
